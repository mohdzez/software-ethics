The thesis

Computer failures are rarely single random events.
They are chains.

A human mistake, a design assumption, a hidden fault, a weak process, a missing safeguard — each propagates to the next until it becomes a visible failure and real-world harm.

Speak precisely

Five words people use as if they were one

TermWhat it really meansConcrete example
MistakeA human misunderstanding or wrong actionA developer misreads the requirement
Fault · bugA hidden flaw left inside code, hardware, design or configA bad unit conversion; an overflow waiting to happen
ErrorThe incorrect internal state when the fault is activatedThe system now holds a wrong value
FailureThe wrong behaviour, now externally visibleApp crashes · rocket veers · payment fails
IncidentA failure with operational, social or business impactFlights cancelled · hospitals delayed · money lost

Core model — fault → error → failure — formalised by Avizienis, Laprie & Randell (2004). A fault can lie dormant for years; it turns dangerous only when activated, and becomes a failure only when the wrong state crosses the service boundary. We bracket it with the human mistake before and the operational incident after.

🦟 The first “bug” was literal — a moth taped into Harvard’s Mark II logbook, 1947: “First actual case of bug being found.”

Reframe reliability

A reliable system is not one that never has bugs.

It is one where bugs are expected, contained,
detected and recovered from

01
A researcher's toolkit

Three ways to
classify a failure

By its cause, by its duration, and by its behaviour. Using more than one axis is what makes analysis feel rigorous.

Axis A · by source & cause

Where the flaw is born

Hardware

A physical component fails — disk crash, RAM error, overheating.

Software

Logic behaves wrongly — overflow, race condition, memory leak.

Network

Connectivity, routing or DNS breaks — BGP outage, packet loss.

Human / process

Bad deploy, weak review, poor comms — wrong config to production.

Security / supply-chain

A trusted update becomes the failure source — CrowdStrike.

Interface / integration

Two systems disagree on units, format or protocol — Mars Orbiter.

Axis B · by duration

Transient

Temporary; disappears on its own — a network glitch.

Intermittent

Comes and goes; hard to reproduce — a loose connector, a race.

Permanent

Stays until repaired — a dead disk, a broken release.

Axis C · by behaviour

Crash / fail-stop

Stops completely — BSOD, kernel panic.

Omission & timing

No response, or a response too late — a server timeout, a missed deadline.

Value & Byzantine

Wrong output, or inconsistent & conflicting outputs across nodes.

Most real failures are not only software,
or only human error.

They are socio-technical chains.

Case 01

Blue Screen
of Death

fail-stop · a protective halt
OS / kernel fail-stop crash
STOP
A deliberate kernel halt — not a random glitch
PROTECT
Continuing could corrupt data or break security

A kernel-level driver, device, memory region, file system or service pushes Windows into an unsafe state. Rather than continue and corrupt everything, the OS stops on purpose and preserves diagnostics.

Professional lesson

Sometimes stopping is the safe choice. A good system fails safely, keeps evidence, and helps recovery.

“The BSOD is a failure — but also a safety boundary: the system stops before unsafe continuation does worse.”

Source: Microsoft Learn — “Bug Checks (Stop Code Errors)”

Case 02

Ariane 5
Flight 501

Maiden flight · 4 June 1996  🎂 30 years ago — today

value failure · operand overflow
software reuse inadequate testing
37s
From liftoff to self-destruct
>$370M
Launcher & 4 Cluster satellites lost
64→16
bit float cast to signed int → overflow
2 / 2
Primary & backup failed identically

A horizontal-velocity value — a 64-bit float — was converted to a 16-bit signed integer. On Ariane 5's faster trajectory it overflowed, threw an unhandled exception, and shut down both inertial reference systems. Guidance was lost; the rocket veered and exploded.

Professional lesson

Reusing code is not automatically safe. The software was correct for Ariane 4 — and revalidated for nothing in Ariane 5. The value wasn't even needed after liftoff.

“The bug was not only in the code. It was in the assumption that old code was safe in a new system.”

Source: ESA / CNES Inquiry Board report (mechanism) · cost per press reporting

Case 03

Mars Climate
Orbiter

Lost on arrival · 23 Sept 1999

interface failure · unit mismatch
interface mismatch systems engineering
lbf·s vs N·s
Imperial output read as metric — off by ×4.45
~$327M
Mission cost, lost
−170 km
Lower than planned at Mars insertion

Ground software reported engine impulse in pound-force-seconds. The navigation software read those numbers as newton-seconds. The trajectory error built up silently until the spacecraft passed too close to Mars and was torn apart.

Professional lesson

Assumptions must be explicit. Units, formats, protocols and interfaces are not small details — they are contracts.

“It was lost not because engineers couldn't calculate — but because two systems never agreed on what the numbers meant.”

Source: NASA — Mars Climate Orbiter mission findings

Case 04

CrowdStrike
Global Outage

19 July 2024

supply-chain · out-of-bounds read
supply-chain unsafe rollout
8.5M

<1%
of all Windows machines — yet global

the bad file was live
21 ≠ 20
Rule read a 21st field; only 20 supplied

A security-content update slipped past a faulty validator. A new rule then tried to read a 21st field when the sensor only ever supplied 20 — an out-of-bounds read deep in the kernel. Every machine that pulled the file hit a BSOD — from a file that was live for just 78 minutes.

Professional lesson

The percentage can be tiny — but if the affected systems are critical, the social blast radius is enormous. Airlines, banks and hospitals stopped.

“Modern failure is about blast radius: one trusted update can ground airlines, banks, hospitals and governments.”

Source: CrowdStrike Post-Incident Review · Microsoft outage update

Optional · Ethics

Therac-25

Radiation overdoses · 1985–1987

race condition · removed safeguard
safety-critical weak response
≥ 6
Massive overdoses; deaths & severe injury
~100×
Over the intended therapeutic dose
8 s
The edit window that triggered the race

A race condition — triggered when an operator edited the prescription quickly — let the machine fire a massive dose. Software was reused without independent verification, and the hardware interlocks that protected earlier models had been removed.

Professional lesson

In safety-critical systems, software must not be the only line of defence. Interlocks, risk analysis, and taking user reports seriously are ethical obligations.

“When software controls physical harm, “bug” becomes an ethical word.”

Source: Leveson & Turner — investigation of the Therac-25 accidents

Optional · Finance

Knight
Capital

45 minutes · 1 August 2012

deployment failure · no stop condition
deployment failure automated trading
$460M
Lost in ~45 minutes
4M+
Executions, to fill just 212 orders
397M
Shares traded by accident
$12M
SEC penalty · firm nearly bankrupt

A new release wasn't deployed to one of eight servers. A reused flag woke dormant legacy code from 2003 on it — and orders fired with no stop condition: roughly $10 million a minute. 97 automated warning emails went out before the open. None were read.

Professional lesson

Fast automated systems need kill switches, deployment discipline, monitoring and risk controls — before they run, not after.

“Automation increases speed — including the speed of failure.”

Source: U.S. SEC — Knight Capital Market Access Rule order (2013)

The pattern across all of them

Same chain, different costume

🔁 A reused assumption

Old code, trusted in a new context it was never validated for. Ariane 5.

🔌 An unspoken interface

Two parts disagreeing on meaning — units, fields, protocol. Mars · CrowdStrike.

⚙️ A weak process

A deploy, a validator, a review that didn't catch it. CrowdStrike · Knight.

🛡️ A removed safeguard

The last line of defence taken away. Therac-25.

In every case, there was a moment where the chain could have been broken.

03
What professionals actually do

You can't delete bugs.
You can break the chain.

We will never remove every fault. So professional practice is about stopping fault → error → failure → incident from propagating to the end.

Defense in depth · the Swiss-cheese model

Every layer has holes. Stack enough layers.

📋 Requirements clarity
catches wrong assumptions
🧩 Design & code review
catches architectural & hidden defects
🧪 Testing · fuzzing · fault injection
catches edge cases & bad input — would have caught CrowdStrike
🔗 Interface contracts & simulation
catches unit/protocol mismatch — Mars, Ariane
🐤 Canary & staged rollout
limits blast radius — would have contained CrowdStrike & Knight
📈 Monitoring · rollback · kill switch
detects silent failure & recovers fast
Learn, don't punish

Blameless
postmortems

After every significant failure: write down what happened and why — without indicting a person. Blame teaches people to hide problems. Learning teaches systems to survive them.

Source: Google SRE — Postmortem Culture

❌ Blame culture

"Who broke it?" → people stay quiet → the same failure returns.

↓

✅ Blameless culture

"What let this happen?" → causes surface → the chain gets a new barrier.

The professional's duty · ACM Code of Ethics

Quality is not a preference. It is a duty.

🎯 High quality in process and product

Not just shipping — building it well, and reviewing how it was built.

⚠️ Comprehensive risk evaluation

Identify and analyse the ways your system could cause serious harm.

🔒 Robust, usable security

Design for the failure and the attacker, not just the happy path.

🌍 Respect those affected & the public good

The people downstream of your code never signed off on your assumptions.

The takeaway

Systems don't just inherit our code.
They inherit our assumptions.

Professional practice means making those assumptions visible, testing them, reviewing them, and designing recovery — before users ever pay the price.

Remember the chain.

Six more for the record

1994 · hardware

Pentium FDIV

A division lookup table missing 5 entries returned wrong results. Intel's first recall — a $475M charge.

1990 · cascade

AT&T collapse

One misplaced break in recovery code rippled through 114 switches. ~50% of calls failed for 9 hours; >$60M.

2003 · race condition

Northeast Blackout

A race condition stalled GE's XA/21 alarms — operators flew blind. ~55 million people lost power.

2018–19 · safety

Boeing 737 MAX

MCAS trusted a single angle-of-attack sensor and pushed the nose down. 346 lives lost.

2000 · dormant fault

Y2K

A 2-digit-year fault, dormant for decades. A massive prevention effort meant the failure mostly never happened.

2014 · security

Heartbleed

A missing bounds check in OpenSSL let attackers read server memory — a fault that was also a global security incident.

The fix Ariane needed

Redundancy,
done right

Two identical backups fail identically against a deterministic bug — that's why Ariane 5's second unit died 0.05s after the first. The answer is dissimilar redundancy: independent teams, independent implementations, so they can't share the same fault.

🚀 Space Shuttle

Four identical flight computers plus a fifth running separately-written Backup Flight Software — a different team, a different program.

✈️ Modern fly-by-wire

Boeing 777 & Airbus use dissimilar hardware and software channels, so a single coding fault can't take them all down at once.

Redundancy only helps if the copies can fail differently.
